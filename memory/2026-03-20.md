## Snow-Flow MCP Bug & Fix (2026-03-20)

### Problema detectado
La herramienta `snow_order_catalog_item` del MCP Snow-Flow crea órdenes de catálogo (RITM) pero **NO** crea los valores de variables en `sc_item_option`. Solo crea registros en `sc_item_option_mtom` con campos incorrectos.

**Symptoms:**
- RITM se crea exitosamente.
- `sc_item_option_mtom` tiene registros pero el campo `item_option` está vacío (debería ser `sc_item_option`).
- `sc_item_option` queda vacío => no se pueden leer valores de variables después.

**Root cause:**
El código en `snow_order_catalog_item.ts` envía `name` y `value` a la tabla `sc_item_option_mtom` (que es solo para relaciones) en lugar de crear registros en `sc_item_option` y luego relacionar.

### Implementación corregida (snowflow skill)

Se modificó la skill personalizada `skills/snowflow/snowflow.js` para usar una implementación manual que garantiza el proceso completo:

1. Crear `sc_request`
2. Crear `sc_req_item` (RITM)
3. Consultar `item_option_new` (definiciones de variables) para el catálogo
4. Para cada variable proporcionada:
   - Crear `sc_item_option` con `item_option_new` (sys_id de la definición) y `value`
   - Crear `sc_item_option_mtom` con `request_item` (sys_id del RITM) y `sc_item_option` (sys_id del recién creado)

**Corrección clave:** En `sc_item_option_mtom` el campo correcto es `sc_item_option`, no `item_option`.

### Validación exitosa

- **RITM0010068** creado con variables `first_name`, `last_name`, `personal_email`.
- Se crearon 3 registros en `sc_item_option` con sus valores.
- Se crearon 3 registres en `sc_item_option_mtom` vinculando correctamente al RITM.
- La función `getRITMVariableValues` ahora devuelve los valores correctamente.
